## 1. 전체 화면 구조 (IA)

### 공통 흐름

```
[스플래시]
  → [운영방식 선택]
      → (심플형)
          → [정책 설정]
          → [홈]
      → (매니저형)
          → [학교 선택]
          → [학년 선택]
          → [반 선택]
          → [학생 정보 입력]
          → [홈]

[홈]
  → [탐지 기록 목록]
      → [탐지 상세]
  → [설정]
```

## 3. 핵심 설계 원칙

1. 화면은 **8개 이하**로 제한
2. 모든 상태는 **"홈"에서 확인**
3. 설정/기록 외에는 화면 분기 최소화
4. 텍스트보다 **상태 중심 UI**

```
┌─────────────────────────────┐
│  [상단 상태 영역]              │
│  현재 상태: 🟢 정상            │
│  최근 탐지: 없음               │
│  ─────────────────────────  │
│                             │
│  [중앙 핵심 카드]              │
│  ┌─────────────────────┐    │
│  │    현재 단계          │    │
│  │   누적 탐지 횟수       │    │
```

## 9. 노티 UX (핵심)

### 3단계 알림 구조

| 단계 | 레벨 | 스타일 | 메시지 |
|------|------|--------|--------|
| 1단계 | GUIDE | 소프트 팝업 | "주의가 필요합니다" |
| 2단계 | CAUTION | 강조 팝업 | "반복 사용이 감지되었습니다 사용을 줄여주세요" |
| 3단계 | WARNING | 강한 경고 + 액션 유도 | "위험한 사용이 지속되고 있습니다 즉시 중단하세요" |

## 10. 사용자 흐름 요약

### 심플형
```
설치 → 모드 선택 → 정책 설정 → 홈 → 탐지 → 기록
```

### 매니저형
```
설치 → 모드 선택 → 학교 → 학년 → 반 → 번호/이름 → 홈
```

## 11. API 매핑

| 화면 | API 엔드포인트 |
|------|--------------|
| 운영 방식 선택 | `POST /installations/start` → `POST /installations/select-mode` |
| 학교 선택 | `GET /managed/schools` |
| 학년 선택 | `GET /managed/schools/:id/grades` |
| 반 선택 | `GET /managed/grades/:id/classes` |
| 학생 정보 입력 → 완료 | `POST /managed/device-affiliations` → `POST /installations/complete` |
| 홈 (탐지 수) | `GET /detections?limit=100` (클라이언트 집계) |
| 탐지 기록 목록 | `GET /detections?limit=100` |
| 탐지 상세 | `GET /detections/:id` |
| 오탐 신고 | `POST /detections/:id/false-positive` |
| 설정 (소속) | `GET /managed/device-affiliations/me` |

## 12. 중요한 판단

**이 UI 구조의 장점:**
1. 화면 수 최소화 (8개 이하)
2. 학습 필요 없음
3. 모든 기능 홈 중심
4. 플랫폼 통일 가능
